# Button-family styling stays on global SCSS mixins; `loading-indicator` uses `styleUrls`

The library styles its six button-family members (`button`, `icon-button`, `fab-menu`, `fab-menu-trigger`, `button-group`, `split-button`) through global SCSS mixins under `src/lib/styles/components/all-buttons/` that a consumer opts into explicitly (e.g. `@include mat-expressive.mat-expressive-all-buttons-styles()` or the per-component `mat-expressive-button-styles()` variants). `loading-indicator`, which has no button-family membership, instead uses an ordinary Angular `styleUrls: ['./loading-indicator.scss']` with `ViewEncapsulation.None`. This looks like an inconsistency — two different styling mechanisms in one library — but the split is deliberate and follows directly from what each component is and what it needs to share.

**Four of the six button-family members are `@Directive`s, so global mixins are their only option.** `MatExpressiveButton` (`src/lib/components/all-buttons/button/button.ts`), `MatExpressiveIconButton` (`.../icon-button/icon-button.ts`), `MatExpressiveFabMenu`, and `MatExpressiveFabMenuTrigger` (`.../fab-menu/fab-menu.ts`, `fab-menu-trigger.ts`) are all `@Directive`s applied to Angular Material's own `MatButton`/`MatIconButton`/`MatMenu`/`MatMenuTrigger` hosts (`[matExpressiveButton]`, `[matExpressiveIconButton]`, etc.). Angular gives a `@Directive` no template and no `styleUrls` — there is no view for component-scoped CSS to attach to. Each directive's only styling surface is the `host` object contributing `[class]` and `[attr.data-*]` bindings (`data-size`, `data-shape`, `data-state`, `data-toggle`, `data-appearance`, `data-width`, `data-color`) onto the host element Angular Material already rendered. Global SCSS keyed off `.mat-expressive-button[data-size='m']`-style selectors is the only mechanism that can reach that markup. This is forced by the `@Directive` choice, not a preference — a leftover commented-out `@Component({ template: '', styleUrls: [...] })` "Styles" scaffold survives in both `button.ts` and `fab-menu-trigger.ts` from an earlier attempt at a component-scoped path; it was never wired up (`matExpressiveWithStyles` calls are commented out alongside it) and is dead code, tracked for removal separately (issue #104).

**`button-group` and `split-button` are real `@Component`s that *choose* to stay on the same mixin path for consistency, not necessity.** `MatExpressiveButtonGroup` (`.../button-group/button-group.ts`) and `MatExpressiveSplitButton` (`.../split-button/split-button.ts`) each have their own `templateUrl`/`template` and a commented-out (not deleted) `// styleUrls: [...]` line — component-scoped CSS is technically available to them. They deliberately do not use it. Both render their host with the same `[class]` + `[attr.data-size]`/`[attr.data-shape]`/`[attr.data-variant]`/`[attr.data-selection]`/`[attr.data-appearance]` pattern as the directives, and their SCSS (`src/lib/styles/components/all-buttons/button-group/_index.scss`, `.../split-button/_index.scss`) builds the exact same `.mat-expressive-button-group[data-size='#{$size}']`-shaped selectors driven off the shared `config.scss` token tables, `utils/_constants.scss`'s `$known-attributes`, and `utils/functions/_validate-config.scss`. All six button-family members share one combinatorial size/shape/appearance/state/selection/variant token system and must render pixel-identical at every shared value (e.g. `size='m'` must look the same on a lone `button` and on a `button` inside a `button-group`). Forking `button-group` and `split-button` onto component-scoped `styleUrls` would require either duplicating that token system per-component or exporting shared Sass into two styling worlds, splitting the single source of truth the whole family depends on for visual consistency. Staying on the global-mixin path — with `styleUrls` commented out rather than deleted, as a visible marker that it was considered and rejected — keeps one styling mechanism for the whole family regardless of Directive/Component implementation.

**`loading-indicator` has no family to stay consistent with, so it uses the ordinary Angular mechanism.** `MatExpressiveLoadingIndicator` (`src/lib/components/loading-and-progress/loading-indicator/loading-indicator.ts`) is a standalone `@Component` outside the button family, with its own `styleUrls: ['./loading-indicator.scss']` and `ViewEncapsulation.None`. It has no discrete size/shape/appearance variant enum to key attribute selectors off — its SCSS (`loading-indicator.scss`) instead exposes continuous CSS custom properties (`--mat-expressive-loading-indicator-size`, `-min-size`, `-max-size`, `-color`, `-container-color`, `-active-indicator-size`) with fallback defaults inline at each usage site (e.g. `width: var(--mat-expressive-loading-indicator-size, 48px)`). There is no combinatorial token table to keep in sync with sibling components, so there is nothing forcing it onto the global-mixin path. Using `styleUrls` directly means the styles ship with the component and require zero consumer setup — no `@include` call needed — which is the appropriate default for a component with no cross-component visual contract to uphold.

## Considered Options

- **Component-scoped `styleUrls` for `button-group` and `split-button`, global mixins only for the four Directives**: technically possible since both are `@Component`s with the commented-out `styleUrls` line ready to uncomment. Rejected because it would fork the shared size/shape/appearance/state token system in two directions — the same `data-size='m'` value would need independent CSS authored twice (once in a global mixin, once in component-scoped SCSS) to stay pixel-identical, doubling the maintenance surface for every future token change.
- **Convert the four button-family Directives into Components** (wrapping `MatButton`/`MatIconButton`/`MatMenu`/`MatMenuTrigger` internally instead of decorating them) so all six members could use `styleUrls` uniformly: rejected as a much larger API change — it would mean the library no longer augments Angular Material's own button/menu components in place, breaking the `[matExpressiveButton]` attribute-directive usage pattern consumers already depend on.
- **Give `loading-indicator` the same global-mixin treatment as the button family for architectural uniformity**: rejected because `loading-indicator` has no discrete variant enum and no sibling components to stay pixel-identical with — introducing an `@include`-based opt-in step would only add consumer setup friction with no corresponding consistency benefit.

## Consequences

Consumers of any button-family member must `@include` the relevant mixin (`mat-expressive-all-buttons-styles()` or a per-component variant) for styles to apply at all — this is non-optional setup, documented on each component's Styling tab. `loading-indicator` requires no such step; its styles are bundled via `styleUrls` and apply automatically. The two abandoned `@Component({ template: '', styleUrls: [...] })` scaffolds in `button.ts` and `fab-menu-trigger.ts`, and the commented-out `styleUrls` lines in `button-group.ts` and `split-button.ts`, remain in the codebase as historical markers of this decision; issue #104 tracks deleting the two fully-dead Directive-side scaffolds (`button.ts`, `fab-menu-trigger.ts`) while the commented-out lines in the two Components stay as intentional documentation of the road not taken. Issue #102 (one source of truth for the TS ↔ SCSS attribute contract) and #103 (parameterizing the six duplicated `validate-*-config` Sass functions) are follow-on cleanups within this same mixin-based mechanism, not alternatives to it.
